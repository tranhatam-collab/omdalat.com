---

# 1. Strategic conclusion

OMDALAT must be treated as a standalone Da Lat platform, not as a branded branch of another active system.

The practical implication is simple:

- OMDALAT owns its own product definition
- OMDALAT owns its own domain family
- OMDALAT owns its own documentation
- OMDALAT owns its own runtime and trust logic
- OMDALAT is judged by real local usefulness, not by alignment to an outside platform

The goal is not to polish a concept page.

The goal is to build a working local platform that helps people, places, hosts, experts, and communities in Da Lat create repeat value through trusted action.

---

# 2. What OMDALAT is

OMDALAT is an independent city-layer platform for Da Lat.

It should help a place become more legible, more trustworthy, and more useful by making real local activity visible and actionable.

OMDALAT is:

- a public explanation layer
- a local opportunity layer
- a trust and proof layer
- a member and operator workflow layer
- a long-term memory layer for real outcomes

OMDALAT is not:

- a tourism brochure
- a generic city portal
- a directory with no trust model
- a shell site that forwards users into another platform
- a branding exercise without operational depth

---

# 3. Core promise

The core promise of OMDALAT is:

**OMDALAT helps Da Lat turn real people, real places, real skills, and real activity into trusted local value.**

That promise should appear consistently across:

- homepage positioning

# 5. Strategic layers

OMDALAT should be built as five connected layers:

## Layer A. Meaning

People must understand the system quickly:

- what it is
- who it is for
- what kind of value it creates
- why Da Lat is the anchor

## Layer B. Opportunity

The platform should surface useful local activity:

- programs
- place-based opportunities
- host offers
- expert capability
- community collaboration
- work and request flows

## Layer C. Trust

The platform should reward credible participation through:

- verification
- proof
- moderation
- completion history
- visible trust signals

## Layer D. Operation

The runtime should support repeat action:

- onboarding
- role routing
- matching
- requests
- profile maintenance
- operator review

## Layer E. Memory

OMDALAT should preserve evidence of real value:

- completed work
- trusted hosts
- active places
- recurring programs
- proof-backed outcomes

# 6. Priority audiences

OMDALAT should primarily serve:

- people who want to live and work in Da Lat with more clarity
- hosts who can open real places and opportunities

- OMDALAT production surfaces must stay inside `*.omdalat.com`
- OMDALAT copy must speak as an independent platform

# 9. Success criteria

The strategy is working when:

- a new visitor understands OMDALAT in under one minute
- docs and public web route users without mixed-brand confusion
- app, auth, and docs feel like one platform family
- the system shows real places, people, activity, and proof
- the repo handoff is clear enough that dev teams can ship without reinterpretation

---

# 10. Final rule

If any strategy, page, or implementation makes OMDALAT look like a dependent shell of another active platform, that work is out of alignment and must be corrected before release.
