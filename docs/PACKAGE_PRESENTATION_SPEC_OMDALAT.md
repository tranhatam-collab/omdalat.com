# PACKAGE_PRESENTATION_SPEC_OMDALAT.md

## Status
Team 1 package communication spec

## Owner
Team 1

## Purpose

This document defines how package information is communicated on public web and public docs.

It does **not** define runtime unlocks.
That remains Team 2 ownership.

## Core rule

Public surfaces may show:

- package names
- role-fit explanation
- comparison language
- upgrade timing guidance

Public surfaces may **not** show:

- exact pricing
- runtime entitlements as source of truth
- unlock claims that Team 2 has not confirmed

## Public package ladder

### Shared entry

- `Free Member`

### Member lane

- `Free Member`
- `Member Path`
- `Program Access`

### Host lane

- `Free Member`
- `Host Circle`
- `Host Partner`

### Expert lane

- `Free Member`
- `Expert Desk`
- `Expert Partner`

### Program / operator lane

- `Free Member`
- `Program Studio`
- `Operator Access`

## Public messaging rules

- copy should explain who each lane is for
- copy should explain when someone usually upgrades
- copy should explain that exact pricing appears after Free Member
- copy should stay calm and premium, not hype-driven
- copy should never claim that public package text itself creates access

## Required public surfaces

- package overview page
- role-based lane sections
- comparison table by role
- join page links into the right lane
- docs page explaining package visibility rules

## Required gated surfaces

After Free Member, Team 2 may show:

- exact pricing
- runtime package map
- unlock details
- upgrade-ready state

## Handoff requirements to Team 2

Team 1 must notify Team 2 when any of the following change:

- package names
- lane names
- comparison table language
- upgrade vocabulary
- public pricing visibility rules

## Review checklist

- no public exact pricing
- every role has a clear lane
- join flow points to the correct lane
- docs and web say the same thing
- copy does not over-promise runtime truth
