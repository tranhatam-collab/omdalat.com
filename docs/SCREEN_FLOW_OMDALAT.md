# SCREEN_FLOW_OMDALAT
## Version 1.0

---

# 1. Public Web Flow

## 1.1 Homepage

Entry:

- hero
- city snapshot
- core entity sections
- proof section
- join CTA

Paths out:

- places
- hosts
- experts
- communities
- events
- proofs
- join

## 1.2 Listing Flow

Each listing page should follow:

- page intro
- trust or locality guidance
- list or card grid
- filter or sort
- entity detail links

## 1.3 Detail Flow

Each detail page should follow:

- identity block
- summary
- trust state
- related entities
- next action

## 1.4 Join Flow

Join should route users toward:

- participation
- contact
- future app access
- request submission when enabled

---

# 2. Local App Flow

## 2.1 Dashboard

Dashboard should show:

- current snapshot
- pending requests
- trust reminders
- proof reminders
- recently relevant entities

## 2.2 Request Flow

- create request
- review request
- submit
- receive suggested matches
- act or archive

## 2.3 Proof Flow

- upload proof
- link to entity
- submit for review
- see review status

## 2.4 Profile Flow

- edit node
- improve profile completeness
- review verification state
- manage visibility

---

# 3. Admin Flow

## 3.1 Review Flow

- open queue
- inspect entity
- inspect proof or verification
- approve, reject, or limit
- record review note

---

# 4. Cross-Surface Flow

Public detail pages should create a clean bridge into the app where needed, but public browsing must still stand on its own.

The user should always be able to move from:

- proof to entity
- entity to related entities
- event to place
- place to host
- community to event
